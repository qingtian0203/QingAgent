## 【幻灯片 3】先说：本地大模型是怎么跑起来的？

核心概念：量化（Quantization）

- 大模型本质是一个巨大的数学权重文件，动辄几十 GB
- 量化 = 把高精度数字压缩成低精度，文件变小，速度变快，精度小幅损失
- 常见精度：4bit（最小最快）、8bit（质量更好）、16bit（原版）
- Gemma4 26B 4bit 只需要约 12G 内存，普通 M 系列 Mac 装得下

运行框架：Ollama

- Ollama 是一个本地大模型管家：一键下载、一键启动、一键对话
- 就像 Docker 对代码，Ollama 对模型：统一打包、统一运行
- 启动后，它在本机开一个 HTTP 端口，任何程序都可以往这个端口发请求、拿到 AI 的回答

## 【幻灯片 6】osascript 是什么？为什么能控制系统？

AppleScript + osascript：macOS 自带的系统遥控器

- AppleScript 是苹果在 1993 年就内置进 macOS 的脚本语言
- 它的设计目标：让程序之间可以互相控制
- osascript 是在终端里执行 AppleScript 的命令

为什么微信能被它控制？

- macOS 大多数 App 都支持 AppleScript 事件，相当于 App 对外暴露了一些遥控接口
- osascript 可以告诉微信：打开、置于前台、搜索联系人
- 不需要破解任何东西，这是 macOS 的合法系统特性

实际指令流程（概念）：
告诉微信激活自己 → 模拟键盘输入搜索词 → 模拟回车

Python 里一行代码就能调用 osascript，就像打电话一样简单

## 【幻灯片 7A】技术串讲：浏览器指令怎么一步步变成真实操作？

**第一步：浏览器 → Python 服务**

浏览器里有一个 HTML 输入框，用户输入指令点发送，前端 JavaScript 自动向本机发一个 HTTP 请求（POST），把指令文字传到 Python 服务，就和你在网页上提交表单完全一样，只是服务器是你自己的电脑。

**第二步：Python 服务 → Ollama（最关键的一步）**

Ollama 在终端跑起来以后，它不只是一个聊天窗口，它同时在本机的 11434 端口开了一个 HTTP 服务器。你甚至可以直接在浏览器里访问 localhost:11434 看到它在运行。

Python 用 requests 库，把用户的指令包装成一条消息，向 localhost:11434 发一个 POST 请求。这和你用 Python 调用百度翻译 API、调用天气 API 完全一样——只不过这个 API 在你本机，不用联网，不用付钱，不用申请 Key。

请求体里写：用什么模型、系统提示词是什么、用户说了什么。Ollama 把模型的回答以 JSON 格式返回给 Python。

关键理解：Ollama 不是嵌入到 Python 里的，而是两个独立进程通过 HTTP 通信。

**第三步：Python 解析 AI 返回值 → 路由到对应 Skill**

系统提示词里要求 AI 用 JSON 格式回答，例如：{"skill": "WeChat", "action": "send_message", "contact": "张三", "message": "明天开会"}。Python 读取这个 JSON，知道要调哪个 Skill、做什么动作、参数是什么，然后调用对应 Skill 的 execute 方法。

**完整流向（一条指令的旅程）**

你输入「给张三发消息说明天开会」
  → 前端 JS 发 HTTP POST → Python 服务（localhost:8888）
  → Python 构造提示词，发 HTTP POST → Ollama（localhost:11434）
  → Ollama 运行 Gemma4，返回 JSON 意图
  → Python 解析 JSON，路由到 WeChat Skill
  → osascript 激活微信 → 搜索 → 输入 → 发送
  → 截图当前状态，Python 通过 HTTP 返回给浏览器显示

整个链路里有两个 HTTP 调用：一个在浏览器和 Python 之间，一个在 Python 和 Ollama 之间。这两个都是普通的 HTTP，和平时写接口没有任何区别。

Ollama 和 oMLX 的关系：两者都是独立进程，都在本机监听一个端口，Python 只需要改一个 URL 地址就能切换。Ollama 用 11434 端口，oMLX 用 8000 端口，接口格式兼容 OpenAI 标准，换掉就完事了。

## 【幻灯片 7B】第一层：浏览器界面是怎么做出来的？

浏览器显示的页面本质是一个 HTML 文件，由 Python 服务动态生成并返回给浏览器。

整个网页只有三个部分：
- 输入框：用户打字的地方，一个普通的 HTML input 元素
- 发送按钮：点击后触发一段 JavaScript 代码
- 结果区域：用来显示 AI 返回的执行结果和截图

发送按钮被点击后，JavaScript 把输入框里的文字打包成 JSON，用 fetch 函数向本机的 Python 服务发一个 POST 请求。整个前端不需要任何框架，就是最基础的 HTML + JS，也不需要部署，打开浏览器输入 localhost 就能用。

Python 服务用 Flask 或 Python 自带的 http.server 模块监听这个端口。收到请求后，提取出用户输入的文字，交给下一层处理，最后把结果和截图路径打包成 JSON 返回给浏览器，浏览器把结果渲染出来。

## 【幻灯片 7C】第二层：大模型收到请求后做了什么？谁告诉它怎么回答？

Python 收到用户指令后，不是直接把指令扔给大模型，而是先构造一套精心设计的提示词，分两部分：

第一部分是系统提示词（System Prompt），这是写死在代码里的，每次请求都带着它。内容是：告诉大模型它的角色是什么、可以调用哪些 Skill、每个 Skill 支持哪些动作、参数格式是什么、回答必须严格用 JSON 格式输出。

第二部分是用户消息（User Message），就是用户刚才输入的那句话。

大模型读完这两部分以后，按照系统提示词的要求，把用户意图转化成 JSON 格式的结构化指令，例如：
{"skill": "WeChat", "action": "send_message", "contact": "张三", "message": "明天开会"}

大模型在这里扮演的角色不是执行者，而是翻译官：把人话翻译成机器能直接执行的 JSON 格式指令。它不知道也不关心后面怎么执行，只负责理解意图、输出格式正确的结果。

---

## 【幻灯片 7D】第三层：Python 怎么找到对应的 Skill？

Python 拿到大模型返回的 JSON 以后，做三件事：

第一步，解析 skill 字段，知道要找哪个 Skill（比如 WeChat）。

第二步，去 Skill 注册中心查找。项目启动的时候，所有 Skill 都已经提前注册进一个字典，key 是 Skill 的名字，value 是 Skill 的实例。查找就是一次字典查询，O(1) 速度。

第三步，调用 Skill 的 execute 方法，把 action 名称和参数 slots 传进去。execute 方法内部按照 action 名称找到对应的 execute_xxx 方法并调用它，比如 execute_send_message。

整个路由过程没有任何魔法，就是字典查找 + 方法名拼接，Python 最基本的特性。

## 【幻灯片 7E】第四层：Skill 的内部架构——分层设计

Skill 的架构分为两层：

**上层：应用 Skill（按 App 分类）**

每个 App 对应一个 Skill 类，比如 WeChatSkill、BrowserSkill、OSControlSkill（系统控制）、AntigravitySkill（AI 编程助手）等。每个 Skill 负责自己那个 App 的所有操作，互相独立，互不干扰。

每个 Skill 在初始化时注册自己支持的"意图"——本质是一份菜单，告诉 Planner 我这个 Skill 能做什么、需要什么参数、典型用法是什么。

**下层：BaseSkill 基类（共用的底层能力）**

所有应用 Skill 都继承自同一个 BaseSkill 基类，基类封装了所有 App 都需要的通用操作：
- 激活应用（把目标 App 拉到前台）
- 截图（截取应用窗口当前画面）
- find_and_click（截图 → AI 视觉定位 → 点击，三步合一）
- read_content（截图 → AI 读取页面内容）

**最底层：actions 模块（真正的执行手）**

BaseSkill 的底层操作最终全部调用 actions 模块。actions 模块封装了两类工具：
- pyautogui：Python 库，控制鼠标移动、点击、键盘按键，这是模拟人类操作的核心
- osascript：macOS 系统脚本，用于处理 pyautogui 搞不定的场景，比如粘贴中文文字、向微信发送系统级回车键等

所以实际的操作路径是：上层 Skill 调用 BaseSkill 的通用方法 → BaseSkill 调用 actions 模块 → actions 模块用 pyautogui 或 osascript 控制系统。

类比：Skill = 工种（瓦工/电工），BaseSkill = 工具箱（所有工种共用），actions = 具体的工具（锤子/螺丝刀），osascript = 专用电动工具（处理特殊情况）。

## 【幻灯片 10】转机：Ollama vs oMLX，速度差了 10 倍

什么是 oMLX？

- MLX 是苹果自己开源的深度学习框架，专为 M 系列芯片（Apple Silicon）设计
- 充分利用苹果芯片的统一内存架构：CPU、GPU、Neural Engine 共用同一块内存，不需要数据搬运，效率极高
- oMLX 是基于 MLX 的推理服务，接口和 Ollama 完全一样，可以无缝替换

实测数字对比：

- Ollama：单次图片识别约 10 秒，适合纯文字对话、对速度不敏感的场景
- oMLX：单次图片识别约 0.7 秒，适合需要实时图片识别、频繁调用的场景

切换成本极低：代码层面只改一个 URL 地址，其他完全不变

## 【幻灯片 11】有了速度，但定位还是不准——三段式放大定位

问题：模型看的是 2560×1600 的全屏大图，按钮可能只有 20×20 像素，占比不到 0.01%，找不准是正常的

解决方案：三段式放大定位（像用 8 倍镜瞄准）

第一段：全屏截图
  → AI 粗定位：大概在屏幕右上角
第二段：截取目标附近 300×300 的小图
  → AI 细定位：在这个区域的中间偏左
第三段：截取目标附近 80×80 的极小图
  → AI 精确定位：就在这里，误差不超过 3 像素

关键：每一段 AI 看到的图像尺寸相同，但目标占的比例越来越大，识别越来越准

三段加起来总共约 2 秒，一次定位完成

## 【幻灯片 19A】Python 调用本地 AI：具体怎么做？

本地模型暴露的是标准 HTTP 接口（兼容 OpenAI 格式）

- Ollama 启动后，自动在本机开一个端口（默认 11434）
- 你用 Python 的 requests 库，发一个 POST 请求过去
- 请求体里写：用什么模型、发什么消息
- 拿到的回答是普通的 JSON，读出 content 字段就是 AI 说的话
- 总共不超过 10 行代码，这是所有一切的起点

关键理解：本地大模型 = 一个跑在你电脑上的 HTTP 服务

## 【幻灯片 19B】Web 代码写在哪里？怎么调 Skill？

**Web 服务层（Python 写的中转站）**

- 用 Python 的 `http.server` 或 Flask 起一个本地 HTTP 服务
- 前端（HTML 页面）：用户在浏览器里输入指令
- 后端（Python Handler）：接收指令 → 调 AI 解析意图 → 执行对应动作 → 返回结果给前端
- 不需要部署上云，就跑在本机 127.0.0.1 某个端口，浏览器打开即用

**Skill 是怎么组织的**

- 一个 Skill = 一个 Python 类，负责一个 App（微信 Skill / 浏览器 Skill / 系统 Skill）
- 每个 Skill 里注册"意图"：用什么样的用户语言来触发我？
- 再实现一个 execute 方法：触发后具体执行什么操作？
- Planner（AI）把用户说的话翻译成「用哪个 Skill、做什么动作、参数是什么」，自动路由过去

**一次完整调用链（概念）**

用户输入「给张三发消息说下午开会」
  → Planner（AI 解析）：WeChat Skill，动作 send_message，联系人=张三，内容=下午开会
  → WeChat Skill 的 execute_send_message 方法被调用
  → osascript 激活微信 → 搜联系人 → 点进去 → 输入 → 发送
  → 截图当前状态返回给用户确认（HITL 安全机制）

**从 0 到能用，最小路径**

1. 起一个 Python HTTP 服务（20 行）
2. 接收用户输入，发给本地 AI（10 行）
3. 让 AI 回答是否需要执行某个动作（提示词设计）
4. 如果需要，调用 osascript 执行（10 行）
5. 一个最小可用的 AI Agent 就完成了

## 【幻灯片 20】结语

这整件事没有任何魔法。

- 本地大模型 = 一个聪明的 HTTP 服务
- osascript = macOS 自带的遥控器
- 截图 + 识别 + 模拟操作 = 眼睛 + 大脑 + 手

把这三样东西组合起来，就是一个能干活的 AI Agent。
